4. Frontend → Backend call graph

Problem. If the Marketing Dashboard page calls the User Journey Mapping backend, nothing in the system surfaced that relationship. You couldn't see which pages consume which services, or spot orphaned backends with no page coverage.

Change. New scanner builds a bidirectional cap-to-cap call graph.

How it works

  1. For every cap's backend files, parse router.get/post/put/delete/patch('/api/...') registrations → learn which capability owns which API path.
  2. For every cap's frontend files, scan fetch / axios / portalApi calls for string-literal API paths.
  3. Match each frontend cap's referenced paths back to the owner cap (parameterized routes like /api/admin/leads/:id match on segment count + named-param tolerance).
  4. Persist the result on a new capabilities.frontend_calls_capability_ids JSONB column.

What you'll see on the row

Two new chips alongside the B/F/A pillars (suppressed when there's no signal):

↘ uses 3 — this cap's frontend calls 3 other caps' backends. Tooltip lists them.

↗ used by 2 — 2 caps' frontends call this cap's backend. Derived client-side from the forward map.

An empty "used by" chip on a backend cap is a real signal: no page consumes this — either it's orphaned or someone needs to build the dashboard for it.

⚠ Blocker on the call-graph backfill: GitHub anonymous rate limit.

The Accelerator's GitHub connection has no auth token (access_token_encrypted is null). Anonymous GitHub API is capped at 60 requests/hour; this project has ~580 unique files across the backend + frontend caps. The scan exhausts the limit immediately and every readFileFromRepo returns null.

Effect now: column populated for all 138 caps with empty arrays. Chips don't render until a successful scan happens.

Unblock: add a GitHub personal access token to the connection's access_token_encrypted field (5,000 req/hr), then re-run backfillFrontendCallGraph.js. All code shipped; the result will appear immediately on next refresh.
