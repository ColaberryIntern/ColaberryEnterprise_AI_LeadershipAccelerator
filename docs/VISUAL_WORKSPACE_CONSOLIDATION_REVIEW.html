01 What changed (one paragraph)

Lean consolidation pass. No new feature surface. No new architecture. Tightening only.

Critique was promoted to nav position 1 with the bullseye icon. Blueprint moved to position 2 and took on the role of execution surface. The version-labelled "V2" item was renamed to a clean "System". The duplicate "System View" item (the older ProjectDashboard) was dropped from the navigation entirely (its route remains for rollback). The legacy frontend/src/pages/project/VisualReviewWorkspace.tsx received a deprecation header and a top-of-page warning banner that links to the new workspace. Blueprint now picks up ?build=visual-workspace with the compiled prompt staged in sessionStorage and surfaces it as a primary banner (gradient header, dark prompt block, copy / open-in-new-tab / back-to-Critique / dismiss actions). Inside the Critique workspace, every "Build Center" label was renamed "Blueprint". Frontend tsc exit 0; backend untouched.

Critique & notes — section 2

03 The unified loop

Per spec: "Critique → Prompt Generated → Routed Into Blueprint Build Execution → Verification → Visual Workspace Review. That is ONE loop."
1. Pin
/portal/visual-workspace
2. Compile
Critique workspace
3. Execute
/portal/project/blueprint?build=visual-workspace
4. Verify
Blueprint flow + Critique re-render
5. Resolve
/portal/visual-workspace (mark resolved)

What enforces it

  • The Critique workspace's Send to Blueprint button stages the prompt in sessionStorage and navigates to /portal/project/blueprint?build=visual-workspace.
  • Blueprint reads the param + storage on mount and renders a primary banner above its existing flow (see §5 mockup).
  • After the build completes, the user returns to Critique via the back link in the banner; pin states reflect the rebuild on iframe reload.
  • "Mark resolved" lives only inside Critique — Blueprint is execution, not state-of-truth for critique resolution.

04 Blueprint as execution surface

Per clarification: "Blueprint is still correct. BUT: Blueprint should become the execution orchestration layer."

Blueprint now answers three kinds of execution requests, not one:

OriginHow Blueprint receives itVisual treatment
Cory's plan / autonomous mode Existing flow (no change) Existing Cory plan UI
Capability "Build" buttons Existing per-capability prompt flow (no change) Existing prompt panel
Critique workspace handoff (new) ?build=visual-workspace param + sessionStorage:visualWorkspace:pendingBuildPrompt Primary gradient banner above the page (see §5)

Crucially: the Critique handoff does not bypass Blueprint's existing flow. It surfaces the prompt prominently, but the operator still copies it to Claude Code and runs it manually, same as any other Blueprint-issued prompt. One execution lane preserved.

13 Open questions for the user

Decisions that are within the consolidation philosophy but need your call.
  1. Order of items 2–3 in the new nav. Currently Blueprint at 2, System at 3. Should "System" be ahead of Blueprint as a more natural "Overview"? My read: Blueprint at 2 because the new loop puts execution right after critique; "System" is reference, not workflow. Confirm or flip?
  2. Hard-delete the legacy VisualReviewWorkspace.tsx? The deprecation banner is in place but the file still ships. We could delete it now to remove dead weight, or wait the spec-suggested period.
  3. Hard-delete the /portal/project/system route? Same call — nav-hidden today. Delete the route too, or wait for zero-traffic confirmation?
  4. "System" label. Other reasonable names: "Overview", "Map", "Architecture". Does "System" feel right?

  6. Should Critique be the default landing when the user clicks "Accelerator Portal" in the navbar (currently lands on Blueprint via /portal/project/blueprint)? Pure brand decision — do operators "wake up to" Critique or Blueprint?
