**govContractsTurnWatcher — fix duplicate "[Your Turn]" email storm (idempotency + coalescing + lock).**

2026-06-16
What changedAli received ~10 identical `[Your Turn] UTD Residential Life - BUILD` emails (same bid, same next task, same 6:20 AM timestamp). Root cause in `backend/src/scripts/govContractsTurnWatcher.js`: dedup state (`tmp/ops-engine/gov-turn-state.json`) was written ONCE at end-of-run while the email fired mid-loop, so any crash after a send lost the checkpoint and a retrying scheduler re-sent the same handoff every cycle; separately the inner loop fired once PER new completion, so N completions on one bid = N emails for the same next task. Rewrote with: (1) per-handoff idempotency key `(bidId, nextTaskId)` stored in `state.firedKeys` — a sent handoff never re-emits even if its completion is re-detected; (2) crash-safe checkpoint — state is persisted immediately after a successful send, not at end-of-run; (3) a TTL lock file (`gov-turn.lock`) so concurrent invocations can't both fire; (4) one turn per bid max regardless of completion count; (5) per-tick COALESCING — all waiting bids bundle into ONE joint email (single bid keeps the old per-bid subject; multiple → `[Your Turn] N Gov Contracts bids waiting on you`), the consolidation Ali asked for; (6) `advanceSeen` skips bids whose send failed so they retry; (7) `pruneFiredKeys` bounds the dedup set at 30 days. Pure decision logic (`computeTurns`, `selectNextOverall`, `advanceSeen`, `recordFired`, `pruneFiredKeys`, `renderTurnsEmail`, `classify`, `turnKey`) extracted and exported behind a `require.main === module` guard.
Verification`npx jest src/scripts/__tests__/govContractsTurnWatcher.test.ts` → 19/19 pass, including explicit defect-guard cases (already-fired handoff does not re-emit; N completions → one turn; re-detecting a fired completion after `recordFired` yields zero turns; failed-send bid left un-advanced for retry; no em-dashes survive rendering). `node --check` passes; requiring the module loads only the pure helpers (does not fire Basecamp/Mandrill).
NotesIdempotency violation per CLAUDE.md > Idempotency & Replayability (NON-NEGOTIABLE) — the "works once but re-sends on retry" class. The script is a host-cron operational script (not container code), so the fix takes effect on the next invocation without a deploy. New state field `firedKeys` is additive; the old `lastSeenCompletions` is still honored and `loadState` backfills both keys, so existing state files upgrade in place.
