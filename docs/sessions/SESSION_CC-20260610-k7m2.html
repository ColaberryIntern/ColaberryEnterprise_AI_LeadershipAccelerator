**Family Command Center V2: briefing is now data-driven (live Google Calendar + Gmail) instead of re-sending baked Jun-9 content with an advancing date.**

2026-06-10
What changedAli flagged that today's (Wed Jun 10) briefing header read "Wednesday, June 10" but Section 1 was still labeled "Tuesday" with Jun-9 content — the data was never refreshed. Root cause: the V1 renderer baked all 9 sections as static content; only the date chrome (banner/hero/footer) advanced off `new Date()`, so every daily cron re-sent the same Jun-9 snapshot with a fresh date label. **Rewrote `backend/src/scripts/lib/renderFamilyBriefingEmail.js` to be 100% data-driven** — it now consumes a `briefingData` object (documented CONTRACT in the file header) and every section renders from data with honest empty states; zero baked content, so stale-content-with-fresh-date is structurally impossible. **New `backend/src/scripts/lib/compileFamilyBriefingData.js`** — the live compiler: Google Calendar (service account, impersonating ali@colaberry.com) → today's family events, 7-day week grid, travel cards from upcoming multi-day events, and cross-calendar conflict detection (family event overlapping a timed work event); Gmail (OAuth as ali@colaberry.com, which receives the Hotmail/Procare forwards) → "new since yesterday" from forwarded school/Procare + travel confirmations (uses message `.snippet`, no body parsing). Hero/actions/risks are derived from the above. Each source is independently try/caught: a failure or missing config records the source name in `meta.degraded` and that section renders empty. CT-correct day bucketing (no tz lib; offset computed via `Intl`). Gmail kept gentle (maxResults 8, single 429 backoff-retry, metadata-only) because the creds are shared with the prod inbox-sync. `googleapis` is required lazily inside the client builders so a host env missing the package degrades the source instead of crashing the host-side cron. **New `backend/src/scripts/discoverFamilyCalendar.js`** — read-only; enumerates the calendars the service account can see (impersonating Ali) and prints the likely "Family" calendarId to set as `GOOGLE_FAMILY_CALENDAR_ID`. **`backend/src/scripts/sendFamilyCommandCenterDaily.js`**: `buildEmailHtml()` now async (compile→render); added a `--preview` flag (compile live, write rendered HTML to /tmp, no send) and a **hard safety guard** — if `meta.degraded` includes `family calendar`, the script BLOCKS the send (no lock written) rather than emailing a hollow briefing. This means until `GOOGLE_FAMILY_CALENDAR_ID` is configured + verified, the daily cron skips cleanly with a log instead of sending stale content. **New `backend/src/scripts/lib/compileFamilyBriefingData.test.js`** — 12 unit tests (category mapping, cross-calendar conflict detection incl. all-day + empty + no-overlap boundary cases, CT day bounds, week bucketing, time formatting, hasFamilyData). **`backend/src/config/env.ts`** + **`backend/.env.example`**: added `googleFamilyCalendarId` / a documented Google Calendar service-account block including `GOOGLE_FAMILY_CALENDAR_ID`.
