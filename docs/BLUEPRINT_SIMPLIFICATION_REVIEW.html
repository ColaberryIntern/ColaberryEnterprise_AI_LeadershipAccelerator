01 What shipped (one paragraph)

One new lean page. Two route changes. One legacy banner.

A new frontend/src/pages/project/ExecutionLane.tsx ships as a focused ~520-line page implementing the 6-step execution flow (Context → Task → Prompt → Execute → Verify → Iterate). It reads operational state exclusively from useUnifiedProjectState and consumes Critique handoff prompts from the existing sessionStorage:visualWorkspace:pendingBuildPrompt contract. The route /portal/project/blueprint now serves ExecutionLane; the legacy SystemBlueprint (2,026 lines) was moved to /portal/project/blueprint-legacy for rollback safety, with a top-of-page warning banner pointing operators to the new surface. Mutations (/next-action/accept, /next-action/complete, /verify) reuse existing endpoints unchanged. Backend untouched. Frontend tsc exit 0. Zero new dependencies.

"Blueprint no longer thinks. Cory thinks. Blueprint executes."

What each step does

#StepSource / Action
1Context Reads state.next_action.title + .reason + meta. If Critique handoff present, shows the page route.
2Task Reads next_action.metadata.files_suggested + requirement_key + action_type. For Critique handoff, says "see prompt for full spec."
3Prompt Critique handoff prompt OR a lightweight template built from the action. Copy / open-in-new-tab / mark-accepted buttons.
4Execute Textarea for the validation report from Claude Code. Submit calls POST /api/portal/project/verify.
5Verify Shows the verify result (pass/fail) OR an idle view with project health % + recent verification pass rate from state.health.
6Iterate "Complete & continue" calls /next-action/complete + refresh(). "Back to Critique" links to /portal/visual-workspace. "Open Home" link.

Step 6's "Complete & continue" button is disabled until Step 5 verifies (when there's a Cory action to complete). For Critique handoffs (no Cory action behind them), Complete just dismisses the handoff and refreshes. No friction.

09 Authority preservation

Cory's authority is enforced by structure, not by reminder banners.
  • No local prioritization. The page renders state.next_action verbatim. There is no rank, no sort, no recommendation logic.
  • No alternative actions. The page does not offer "or pick something else from the queue" — for that, the user goes to Home.
  • No local readiness/coverage. Step 5 reads state.health.score and state.health.verification_pass_rate only.
  • No autonomous mode. The page never executes anything for the user; Claude Code is always run by the operator.
  • One mutation lane. Accept / verify / complete only — no other writes happen from this page.
  • Refresh after mutation. Every successful mutation calls refresh() on the unified hook so Home + Critique + everywhere see the new state.

10 What this sprint did NOT do (intentional)

Honest scoping.

Out of scope this sprint

  • Did NOT delete SystemBlueprint.tsx. The legacy surface is preserved at /blueprint-legacy with a warning banner.
  • Did NOT migrate the Components Grid, Project Setup Wizard, Demo Overlay, or System Prompt editor. They remain on the legacy route until they have proper homes.
  • Did NOT touch backend endpoints. The new page calls existing /next-action/accept, /complete, /verify.
  • Did NOT introduce new state machines. ExecutionLane has 4 useState hooks vs SystemBlueprint's 15+.
  • Did NOT add backend cognition / governance / orchestration of any kind.
  • Did NOT add autonomous execution. The operator always runs Claude Code by hand.
  • Did NOT remove the navItem "Blueprint". Same name, new home.

13 Open questions for the user

Decisions that shape the next sprint.
  1. Legacy deletion timing. SystemBlueprint stays at /blueprint-legacy. Hard-delete after 1 week of zero-traffic, 1 month, or wait for explicit "ship it" call?
  2. Components Grid future. Should the capability grid move to /portal/project/system-v2 (System View) as a topology-only display (no build buttons), or get its own dedicated route?
  3. Project Setup Wizard. Currently buried inside SystemBlueprint. Should it move to /portal/project/setup as its own lean route, or stay where it is?
  4. Demo flow. The 60-second demo overlay is a sales asset. Should it become its own /portal/demo route, or be triggered from Cory Home as a "tour" affordance?

  6. Step 5 idle view. Currently shows project health % + recent verification pass rate. Useful, or noise that competes with Home?
  7. Skip affordance. Add a quiet "Pick something else" link in Step 3 (links to Home) for operators who don't want to execute Cory's pick?
  8. Visual workspace handoff completion. When the operator marks Critique-driven build complete, should they auto-redirect back to Critique to mark the issue resolved, or just refresh and stay on ExecutionLane?
