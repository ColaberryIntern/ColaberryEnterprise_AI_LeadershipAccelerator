Chapter 2 · The architecture

One flow, drawn end-to-end. Bottom-up reads like a Lego instruction.

Each employee's laptop talks to Claude Code. Claude Code reads canonical skills from the AI Project Architect GitHub repo. When the employee invokes a skill, three approval gates sit between intent and any external system. Every action is logged. Here is the picture.

Layer 1 · Identity

The 4 employees, each on their own laptop

Karun, Kes, Akiwam, Obi sign in to Claude Code with their own credentials. Their personal Basecamp OAuth refresh tokens, Gmail tokens, Bonfire tokens are stored locally in ~/.colaberry/credentials.json. Their access to your APIs follows their existing role in each system.
↓
Layer 2 · Tool

Claude Code, configured per person

A starter .claude/ folder shapes what their Claude can do. Per-user CLAUDE.md sets their persona. .claude/settings.json tool allow/deny lists. .claude/hooks/ intercept high-risk actions for confirmation.
↓
Layer 3 · Source of truth

AI Project Architect GitHub repo (canonical skills + use cases + agents)

Every Colaberry-approved Use Case, Skill, and Agent lives here. Each employee's .claude/skills/ auto-syncs nightly from this repo. Personal local skills are allowed but do not replicate. When you flag a new skill "Colaberry approved" in the library, it lands here as a Markdown file - everyone gets it on next sync.
↓
Layer 4 · Approval gates (three of them, stacked)

Default prompt mode · Hooks · Plan mode

Gate A: default permission mode prompts before every tool call. Gate B: hooks intercept high-risk actions (BC writes, email sends, deploys, deletes) and require typing a one-line reason + "yes." Gate C: for big tasks, the agent enters plan mode, produces a written plan, the user reads + approves before any tool fires.
↓
Layer 5 · External systems

Basecamp · Gmail · HubSpot · Bonfire · GitHub · GHL · CCPP · Telemetry

The actual systems. Each call is logged with: timestamp, employee, skill used, what was sent, what came back, which gate approved it. The log flows into Cory's daily digest to you.
Nothing executes against an external system unless the employee personally said "yes" to the action, OR the action is on the explicit auto-allow list for their role. There is no autonomous magic. The system is loud about asking.
Chapter 3 · The four security layers

Defense in depth. Any one of these would catch a bad action; we stack four.

1
Identity
Each person has their OWN credentials. Akiwam cannot accidentally do something as Karun because she does not have Karun's tokens. Bonfire credentials never leave the laptop they were set up on.
2
Authorization
Their existing system roles enforce permissions BEFORE Claude even tries an action. Akiwam can read Bonfire but cannot post a bid (Bonfire role says so). Karun can comment on his BC projects but cannot delete the company; Basecamp role says so. No Claude code change can override this.
3
Approval gates
Even on actions they are authorized to take, Claude Code asks. Default mode prompts on every tool call. Hooks add extra friction on high-risk actions (send email, deploy, delete). Plan mode adds a written plan + sign-off for big multi-step tasks.
4
Audit
Every action gets logged - who, when, what skill, what input, what output, which gate approved. Cory rolls this up to you nightly. Anomalies (deletes, mass changes, off-pattern behavior) page you immediately. Nothing happens without a trace.

Chapter 6 · The Bonfire / gov-contracts surface (special handling)

Why Akiwam + Obi get a knife-edge surface, not the full kitchen.

Gov contracts have stakes other workflows do not. A submitted bid is an irreversible commitment. A missed clause in a capability statement is a compliance issue. A confidentiality breach on RFP content is a contract violation. Interns get a sandboxed surface so the upside (faster discovery + better fit scoring) is captured without the downside (someone clicks Submit on a bad bid).

What Akiwam + Obi CAN do

  • Read both Bonfire accounts (Que-shared + Colaberry-only)
  • Search, filter, and triage opportunities (≥10 days out, exclude used UUIDs)
  • Generate fit-score cards against the capability statement
  • Draft bid responses (text only - not the final document)
  • Post comments + recommendations on the Gov Contracts BC list
  • Tag Vinay or David Lahme on findings
  • Pair-review each other's drafts before passing to Vinay

What they CANNOT do (hard blocks at the tool layer)

  • Submit any bid via Bonfire - escalation to Vinay + Ali required
  • Send any external email from any account - tool-layer denial
  • Modify the capability statement - read-only access
  • Access customer data on HubSpot or CCPP - no token issued
  • Touch finance, HR, or any other team's BC project - role-locked
  • Write to GitHub repos other than the gov-contracts pull-only workspace - tool-layer denial
The deeper why: the goal is to give Akiwam + Obi enough surface to learn the trade (RFP comprehension, bid/no-bid reasoning, capability-statement craft) without letting a single click of theirs commit the company to a bad outcome. Vinay still owns every submitted bid. The interns accelerate Vinay; they do not replace Vinay.
