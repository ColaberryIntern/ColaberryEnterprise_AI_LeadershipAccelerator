Operational Leverage + System Influence — Sprint Review

Shipped 2026-05-15 to enterprise.colaberry.ai (commit 4a98938). The previous sprint taught the workspace to communicate where the operator is shaping the system. This one teaches it to communicate where in the operational system effort would ripple furthest — without recommending anything.

The system has the structural facts. Some domains support many others; some have room to mature; some are constrained by upstream gaps. Reading those facts editorially produces leverage — a calm observation about where effort might travel furthest. This sprint surfaces that observation. It never tells the operator what to do.
What this sprint explicitly is NOT: a recommendation engine, a task inbox, an AI copilot, productivity software, workflow automation, prescriptive PM tooling, or anything that ranks tasks for the operator. The anti-prescription guardrails — no imperatives, no certainty words, conditional verbs only — are asserted by the unit test suite.
leverageScore = downstreamCount × maturityHeadroom
A foundational domain that supports four others (headroom 5 × 4 = 20) outscores a stabilized anchor with the same downstream (0 × 4 = 0). The system calls out where effort has room to ripple — not what the operator should do.

1The leverage headline — system-level reading

Above the domain stack on System BPs, a calm editorial sentence reads the structural facts and names where effort would travel furthest. Conditional verb ("would unblock"), explicit downstream count, named downstream areas. Silent when no domain meaningfully stands out.

Operational leverage headline + system-evolution sub-line · production · 2026-05-15
What the headline says:
  • Top line — names the highest-leverage domain + the conditional ripple ("strengthening it would unblock 3 downstream areas (…)").
  • Italic sub-line — overall system evolution phrase ("Your operational system is still being scaffolded — most areas have room to mature.").
  • Three reason phrasings based on the score profile: constrained_downstream ("strengthening it would unblock N"), broadest_surface ("ripple through"), mature_anchor ("currently anchors the broadest operational surface", no "would" — contextual).
  • Threshold: headline only fires when leverage score ≥ 3. Below that, silent.

2Per-domain forward-looking note — the mirror of pressure

Each domain row already carries a backward-looking pressure note (amber italic) when an upstream area is weaker than it. This sprint adds a forward-looking leverage note (blue italic) when there is room downstream. Two halves of the same coin — pressure looking upstream, leverage looking downstream — living in the same italic group so they read together.

Forward-note language ladder (calm, conditional, gated on maturity):
  • Foundational / Emerging (low maturity, ≥1 downstream): "Strengthening this would stabilize the N downstream area(s) that depend on it."
  • Coordinated / Operational: "Continued maturation here would reinforce N downstream area(s)."
  • Scaling: softer — "Further refinement here may further stabilize N downstream area(s)."
  • Stabilizing: no headroom → no note (silent).
  • Zero downstream: no note (silent — nothing forward to say).

4Honest framing — observation, never recommendation

The most psychologically loaded line in the sprint is the leverage headline. It would be very easy to drift toward "Focus your effort on Lead Intelligence" — a recommendation. Instead, every sentence describes structural facts about the system with conditional verbs, leaving the operator to draw the inference.

Forbidden framingShipped framing
"You should focus on Lead Intelligence next.""Highest operational leverage currently sits in Lead Intelligence — strengthening it would unblock 3 downstream areas."
"Fix Intake first — it's blocking everything downstream.""Strengthening this would stabilize the 2 downstream areas that depend on it."
"Optimal next move: stabilize Execution.""Continued maturation here would reinforce 1 downstream area."
"Lead Intelligence will guarantee Reporting improvements.""Lead Intelligence currently anchors the broadest operational surface."
RecommendationObservation
How this is enforced:
  • All leverage language lives in one file: operationalLeverage.ts.
  • Unit tests assert no imperative phrases anywhere: you should, you must, you need to, fix, address, do this.
  • Unit tests assert no certainty words: guarantee, optimal, perfect, definitely, best practice.
  • Unit tests assert no exclamation marks anywhere in the output.
  • Every builder returns null when there's nothing meaningful to say — surfaces render nothing rather than fall back to filler.

6Overall verdict

The platform now reads leverage from the structural facts it already had on screen — lifecycle states, downstream counts, pressure notes — and surfaces that reading editorially. The headline names where effort would travel furthest; per-row notes mirror upstream pressure with downstream leverage; Home carries the cached observation forward.

Net stats: 9 files (3 new), 211 tests passing (25 new), production deployed and verified. No backend changes, no new endpoints, no recommendation engine. Every leverage statement is a reading of facts the classifier already computes.

What's not in this sprint: task ranking, "next best action" suggestions, AI inference about what the operator should do next. The line we did not cross.
