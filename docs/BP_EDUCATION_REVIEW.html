BP Detail — Education + Cory Deeplinks

Proposal review · target: BPDetailV2 (the modal that opens when an operator clicks a BP row on System > BPs). Operator feedback: the detail panel is "light" — three terse buttons under Next steps, no contextual explanation, and no path back to the Cory assistant for follow-up. This proposal adds an education chip strip (4–6 context-aware questions per cap) and converts the three buttons into a walk-through panel where each step explains what it does, why it matters for this specific BP, and offers a one-click "Ask Cory about this step" deeplink.

Operator wanted "more information on the steps" — not a textbook page glued onto the detail. The shape here keeps the calm one-column layout intact: a quiet chip strip plus richer step cards. Cory remains the deep-dive surface; the detail panel just makes it easy to land there with the right question already typed.

⚙️ Curation rule:
  • Max 6 chips. Priority order: gap-diagnosis → maturity → dependencies → generic.
  • Every question name-drops the BP so Cory has full context. No "this BP" pronoun questions.
  • If a cap has zero gaps (rare — fully-built L4 cap), strip drops to 2 generic chips so it doesn't feel forced.

3Cory deeplink — window event

GlobalCoryWidget already exposes an externalQuery prop that auto-sends a prefilled message. The only piece missing is a way for any surface to set it. Proposal: one global window event.

The API

// Any component, anywhere in the SPA:
import { useCoryAsk } from '../hooks/useCoryAsk';

function MyChip() {
  const askCory = useCoryAsk();
  return (
    <button onClick={() => askCory("What does Lead Classification do?")}>
      💬 Ask
    </button>
  );
}

// Under the hood it fires:
window.dispatchEvent(new CustomEvent('cory:ask', {
  detail: { query: 'What does Lead Classification do?', source: 'bp-detail' }
}));

// GlobalCoryWidget listens, opens itself if collapsed, sets externalQuery.
// CoryPanel's existing externalQuery effect auto-sends.

Why a window event (not context)

  • Zero plumbing through layouts — chip emits, widget listens.
  • Works from anywhere in the SPA: detail, system tab, critique, blueprint.
  • Widget can ignore, debounce, or rate-limit centrally.
  • One line per future caller. useCoryAsk() is the only public surface.

docs/BP_EDUCATION_REVIEW.htmlThis file. Walks through the proposal with mockups + decision matrix.
✓ Deploy plan
  • Frontend-only change. No backend rebuild. Nginx rebuild ≈ 2–3 min.
  • No new dependencies, no new BP fields, no new endpoints.
  • No LLM cost on the BP detail page-load. LLM only fires when the operator clicks a chip (and that already costs whatever Cory normally costs per chat turn).
5What this proposal will NOT do
  • No re-architecture of CoryPanel. It already auto-sends an externalQuery on change. We just wire up a way for any surface to set it.
  • No persistence of which questions the operator has asked. That's chat history Cory already keeps in the panel.
  • No server-side AI-generated questions on the initial load. All chips are derived deterministically from the cap shape. We can add an LLM-curated set later if these feel rote.
  • No new BP fields. If we later want operator-authored educational copy per cap, we'd add a cap.education_notes JSONB column — out of scope for this sprint.
  • No restyling of the Cory widget itself. It opens exactly as it does today.
