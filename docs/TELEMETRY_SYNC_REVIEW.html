Telemetry Sync + BP Categorization — Before/After

Operator caught two related problems: the System surface didn't reflect everything we'd shipped, and most BPs were dumping into "Other Operations." Both are fixed. This doc shows the before/after state of each, the new verification tooling, and the path to closing the remaining gaps.

The portal had been running blind on filesystem heuristics across every recent sprint. The Telemetry Synchronization Contract requires Claude Code to emit a BuildManifest after every non-trivial commit; zero had been emitted. The fix is two new tools — a verifier you can re-run any time, and a backfill emitter that catches the system up.

3Tools shipped — re-runnable any time

Two scripts now live in the repo so this state stays verifiable.

CommandWhat it does
node scripts/verifyTelemetrySync.js Queries 4 production endpoints, counts ground-truth files on disk via Node-native recursive walk (no shell-out — Windows/Mac/Linux all work), reads last 30 git commits, prints a "system thinks vs reality" diff. Exits non-zero when sync is broken — suitable for CI / pre-deploy.
node scripts/backfillBuildManifests.js Walks recent git commits, builds a schema-valid BuildManifest for each (classifies touched files into ui_components_added/modified, apis_modified, tests_added, files_created/modified/deleted), POSTs to /api/portal/project/telemetry. Supports --dry-run, --max N, --since 7.days.
