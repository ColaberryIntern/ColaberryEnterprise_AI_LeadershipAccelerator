## #2 Propose agent stack for **Campaign Management** (agent_stack)

**System framing:** "Campaign Management is built (readiness 80%). Has no agent layer linked. Missing roles: monitor, alert, follow_up. Add agents to cover these roles. ⚠ Roles inferred from filename only (file content unavailable; check GitHub connection auth)."

**Linked files:** 7 backend (adminCampaignController, campaignRoutes, campaignService, Campaign.ts model, campaignSimulationController + routes + model). 3 frontend. **0 agents.**

**What this cap is:** Admin surface for campaign management. Routes, controller, service, model. Customer-facing in the sense that operators manage outreach campaigns through it.

**System's recommendation evaluation:** Real ask — 0 agents on a substantial admin surface is genuinely a gap. The agent stack for this kind of cap would be: monitor campaign send-rate, alert on bounce-rate spikes, follow-up sequences for warm leads.

**BUG FOUND:** The "⚠ Roles inferred from filename only (file content unavailable...)" warning is **misleading** here. The cap has ZERO linked agents — there's nothing to classify, and `agent_roles_cache` is null. But the engine's mapper falls back to `computeCodeEvidence`'s output, which returns `agent_roles: { detected: [], files_inspected: 0 }` even for 0-agent caps. That gets treated as the "filenameOnly tier" and adds the warning.

The warning is intended for "we tried to classify N agents but content wasn't readable." It shouldn't fire when there are no agents to classify in the first place.

**Verdict: REAL ask, but description has the filename-only warning incorrectly attached.** Fix-worthy.

## Bugs / fixes found

1. **`computeCodeEvidence` returns non-null `agent_roles` for 0-agent caps** — causes the misleading "⚠ Roles inferred from filename only" warning to fire for caps with no agents at all (#2 Campaign Management). One-line fix: return `agent_roles: undefined` when `agentFiles.length === 0`.

2. **Brownfield-discovery file classifier mis-buckets `frontend/src/...` files as backend** — surfaces in #4 Validation where the cap's only "backend" file is `frontend/src/services/validationStore.ts`. The classifier in `brownfieldDiscoveryService.classifyFile()` should refuse anything under `frontend/src/` from the backend bucket.

3. **LLM brownfield-consolidation over-merges platform-internal subsystems** — surfaces in #5 (Content Generation + entire OpenClaw under one name) and #6 (Execution Planning = 8 distinct subsystems under one name). Not a queue bug, but the operator's actual next action for these caps is "split before spec'ing," which the system doesn't surface as an option.

## Recommendation

Ship #1 immediately (small, isolated fix). #2 is a brownfield-classifier change that needs careful testing (touches the canonical file-bucketing rules; risk of mis-classifying other files). #3 is a larger change to the LLM consolidation prompt — needs a re-scan to take effect, more effort, and operator should review the new cap structure before locking it in.
