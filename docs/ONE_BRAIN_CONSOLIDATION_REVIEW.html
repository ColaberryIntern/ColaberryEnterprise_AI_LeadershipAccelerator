01 What shipped (one paragraph)

04 UnifiedProjectState shape

The single canonical operational state object. Every surface that needs operational data reads from this. 
interface UnifiedProjectState {
  project: { id, organization_name, industry, project_stage };

  // ONE readiness — every UI surface consumes this
  readiness: {
    score: 0..100;
    band: 'red' | 'amber' | 'green';
    reasons: string[];
    breakdown: { artifact_completion, requirements_coverage,
                 github_health, portfolio_quality, workflow_progress };
  };

  // ONE coverage
  coverage: {
    score: 0..100;
    requirements_matched, requirements_total, bps_complete, bps_total;
  };

  confidence: { score, sources: string[] };
  health:     { score, regressions_24h, verification_pass_rate };

  // THE next step (top of queue) — Cory's authority
  next_action: NextActionProfile | null;

  // ONE ranked operational queue (top 8)
  queue: QueueEntry[];

  // ONE blocker list
  blockers: BlockerEntry[];

  // ONE active build (or null)
  active_build: ActiveBuildProfile | null;

  // ONE verification state
  verification: { pending, passing, failing, pass_rate_24h };

  built_at: string;
}

Source of truth: backend/src/intelligence/unifiedProjectState/types.ts. Frontend mirror in frontend/src/hooks/useUnifiedProjectState.ts.

Critique & notes — section 4

05 UnifiedOperationalPriorityEngine

The ONE ranker. Heterogeneous candidates in → one normalized, deterministically-sorted list out.

Inputs (sources)

  • next_action — top action from nextActionService
  • governance_recommendation — pending governance recs (V1: not wired yet; reserved)
  • visual_workspace_pending — Critique→Blueprint pending builds (V1: not wired yet; reserved)
  • verification_failure — failed checks (V1: not wired yet; reserved)
  • capability_gap — low-completion capabilities (V1: derived from blocker logic)

Sort rules

  1. Primary: descending priority_score (clamped 0..100).
  2. Tie-break 1: source-of-origin nudge (next_action > verification_failure > governance_recommendation > visual_workspace_pending > capability_gap).
  3. Tie-break 2: alphabetical title (deterministic).

Hard rules

  • No surface re-ranks the queue locally. Consumers render in the order returned.
  • The same input always produces the same order (deterministic).
  • Source: backend/src/intelligence/unifiedProjectState/unifiedOperationalPriorityEngine.ts

11 Migration plan for the surfaces still computing locally

Each migration is small and local. Doing them gradually keeps the platform stable while the consolidation lands.
SurfaceMigrationEffort
ProjectNextActionPanel Replace direct fetch with useUnifiedProjectState().state?.next_action; keep render shape. ~30 min
WhyIsThisNextPanel Read same source; explain from state.next_action.metadata. ~1 hour
SteeringPanel + GuidedExecutionPanel Render the unified queue + open in Blueprint; remove local recommendation logic. ~2 hours each
useGovernanceRecommendations Wire governance recs into the synthesizer (queue source governance_recommendation); remove the panel-level hook usage. ~1 hour
useGovernanceAdvice Either: (a) keep as on-demand advice surface (NOT a queue), OR (b) feed into blockers[]. Decision required. ~1 hour
ExecutiveReadinessCard Read state.readiness; remove local getReadiness(). ~30 min
ProjectMaturityGauge Render breakdown from state.readiness.breakdown; keep gauge UI. ~1 hour
RequirementsStatusCard / RequirementsSectionBreakdown Read state.coverage + a coverage-detail endpoint (existing). ~1 hour total

Suggested order: ExecutiveReadinessCard first (smallest, most visible drift), then ProjectNextActionPanel (most user-facing duplicate), then governance migration. Total estimated effort: ~8–10 hours over 1–2 sprints. None requires backend changes.

13 Open questions for the user

Decisions that will shape the next sprint.
  1. Health score. V1 hardcodes health to 90%. Should the next sprint pull from the SystemStateEngine summary (closes the loop with backend cognition) or from a simpler signal (recent test pass rate)?
  2. Reserved queue sources. Wire governance recs / verification failures / visual-workspace pending into the queue this sprint, or wait for Cory Home to settle first?
  3. Migration order. ExecutiveReadinessCard first, then ProjectNextActionPanel? Or take the most-visible duplicate first (ProjectNextActionPanel)?
  4. useGovernanceAdvice fate. Keep it as on-demand advice (separate from the queue), feed it into blockers[], or deprecate entirely?
  5. Greeting personalization. Currently uses organization_name. Should we read the participant's first name (would need a backend tweak)?
  6. Polling cadence. Cory Home polls every 60s. Reasonable, or should we go event-driven via the existing cognitive event bus?
  7. Empty state. When no next_action, Cory Home shows "You're caught up." Is that the right tone, or should it surface a "what could you start" suggestion?
