02 The 5-surface product hierarchy

Five surfaces. Strict ownership. The user's mental model maps left-to-right.

Home

L1 · Authority
"What matters now"
/portal/home

Critique

L2 · Improvement
"What should improve"
/portal/visual-workspace

Blueprint

L3 · Execution
"Run the work"
/portal/project/blueprint

Verify

L4a · Confirmation
"Confirm the result"
(today: inside Blueprint Step 5)

System

L4b · Understanding
"Understand the architecture"
/portal/project/system-v2

Strict ownership rules

  • Home owns: prioritization, queue, readiness, coverage, blockers, active build, verification state.
  • Critique owns: visual annotation, AI suggestions, prompt compilation, issue tracking.
  • Blueprint owns: execution flow only — context → task → prompt → execute → verify → iterate.
  • Verify owns: pass/fail confirmation, screenshots, comparison, acceptance review. Currently embedded in Blueprint Step 5; future sprint may extract.
  • System owns: topology, components, dependencies, advanced visibility. Does NOT prioritize / orchestrate / recommend.

Sessions sits at the right edge of the navbar — orthogonal to the build loop (curriculum/coaching). Not part of the 5-surface hierarchy.

03 Emotional role map

Every surface answers one emotional question for the user.
Surface User feels User does NOT feel Test: if user thinks…
Home Certainty. Calm. Executive. Choice paralysis. "Where do I start?" "What matters today?" — answered immediately by the priority card
Critique Creativity. Iteration. Collaborative with AI. Burdened. Operational. "This page could be better." — pin it, prompt it, ship it
Blueprint Momentum. Guided. Lightweight. Decision overload. "Which button?" "I just need to run this." — paste, verify, continue
Verify Reassurance. Conclusive. Fast. Audit-heavy. Compliance-flavored. "Did it work?" — quick pass/fail confirmation
System Understanding. Curiosity satisfied. Non-blocking. Pressured to act. Decision authority. "What does this look like under the hood?" — explore at leisure

If a surface has conflicting emotional goals, it is fragmented. Cory Home should never feel "operational" (that's Blueprint). Blueprint should never feel "creative" (that's Critique). System should never feel "authoritative" (that's Home).

07 Orphaned-feature placement decisions

The 4 deferred orphans from prior sprints + others called out by the audit, with a one-paragraph decision each.
1. Components Grid ADVANCED
CapabilityGrid.tsx — currently mounted in ProjectDashboard (legacy nav-hidden)
Move to System View as a "Components" tab. Strip the per-card "Build" buttons (no distributed execution). The grid becomes pure topology display: each capability's name, status, completion %, and a link to its detail panel. Build affordances live only in Blueprint via Cory's queue.
2. Project Setup Wizard ADVANCED
ProjectSetupWizard.tsx — currently inline in 3 pages (ProjectDashboard, SystemBlueprint, SystemViewV2)
Extract to a dedicated /portal/project/setup route. Setup is a distinct lifecycle stage (no project yet → onboarding flow); it's not a recurring surface. Cory Home redirects to the setup route when UnifiedProjectState reports no project. Once setup is done, the route is no longer exposed in nav.
3. Demo Overlay ADVANCED
DemoOverlay (inline in SystemBlueprint.tsx) + SystemBuildDemo page
Move to a dedicated /portal/demo route. Surface from Cory Home with a small "Watch 60-second demo" button (CONTEXTUAL — only visible to first-time users or sales-mode triggers). Remove from operator default surfaces.
4. System Prompt editor ADVANCED
Inline in SystemBlueprint.tsx (read/edit autoderived prompt)
Move into the /portal/project/setup route alongside the Setup Wizard. System prompt is a configuration concern, not an execution concern. Once configured, it shouldn't appear on daily surfaces.
5. RequirementsStatusCard / RequirementsSectionBreakdown CONTEXTUAL
RequirementsStatusCard.tsx + RequirementsSectionBreakdown.tsx — currently orphaned
Repurpose as Cory Home Coverage tile detail-on-click + drawer. Both already match the state.coverage shape. Don't delete — reposition. Click the Coverage tile on Home → modal/drawer opens with section-by-section breakdown.
6. ProjectMaturityGauge CONTEXTUAL
ProjectMaturityGauge.tsx — currently orphaned
Repurpose as Readiness tile detail-on-click. Already matches state.readiness.breakdown shape (artifact_completion / requirements_coverage / github_health / portfolio_quality / workflow_progress). Click Readiness tile on Home → opens the gauge breakdown.
7. useGovernanceAdvice surfaces ADVANCED
useGovernanceAdvice hook + any caller
Keep as on-demand utility, surface only from Critique workspace OR from Home blockers section. When advice says should_block_rollout, fold the message into state.blockers[] at the synthesizer level (next-sprint backend tweak). Otherwise advice stays a pull, not a push.
8. Advanced topology visibility (DecisionGraph, ProcessDatabase, ProcessVisual) ADVANCED
DecisionGraphView, ProcessDatabaseGraph, ProcessVisualPanel — currently orphaned
Mount as System View tabs. Three tabs: "Architecture" (Process visual), "Database" (Process database graph), "Decisions" (Decision graph). All ADVANCED — not on the System default tab. Default tab becomes "Components" (the repurposed CapabilityGrid).
9. Archaeology / replay visibility ADVANCED
OperatorCognitionDashboard + AutonomousExecutionDashboard (with 32 phase sections)
Move out of System default surface into "Operations" + "Cognition" advanced tabs. The phase sections (governance memory, handoff continuity, recovery foresight, etc.) collapse into expandable groups within those tabs. Default System view never shows phase numbers.
10. ProjectMentorPanel / Alerts / WorkflowPanel / WorkstationLauncher LEGACY
ProjectMentorPanel.tsx, ProjectMentorAlerts.tsx, ProjectWorkflowPanel.tsx, WorkstationLauncher.tsx
Deprecate. All four predate Cory authority. Mentor and workflow are subsumed by Cory + ExecutionLane. WorkstationLauncher is subsumed by Critique → Blueprint flow. Add deprecation file headers; remove imports; delete after 1 sprint of zero references.
11. ArchitectChat ADVANCED
ArchitectChat.tsx — currently inline in ProjectDashboard
Surface as a Cory chat drawer accessible from Home. Don't make it a primary surface — it's secondary to the operational queue. CoryFullscreen page can become this drawer's "expand" target.

11 Per-surface "should contain / shouldn't contain"

Concrete contract per surface so future placement decisions don't drift.

Home · /portal/home

Should contain
  • Greeting + one-line status
  • Today's One Priority card (Cory authority)
  • 3 tiles: Readiness · Coverage · Health
  • Critical Blockers (CONTEXTUAL — only when present)
  • Operational Queue (top 5–8)
  • Active Build · Verification footer row
  • Optional drawers: Cory chat, Why-is-this-next, Coverage breakdown, Maturity gauge, Steering input
Shouldn't contain
  • Build buttons that bypass Blueprint
  • Capability grid
  • System architecture diagrams
  • Phase numbers / governance jargon
  • Setup wizard
  • Validation report textareas

Critique · /portal/visual-workspace

Should contain
  • Embedded iframe stage with annotation overlay
  • Numbered pins + sidebar issue list (5 sections)
  • Annotation modal + AI suggest pre-fill
  • Right-side details panel + AI suggestions
  • Sticky bottom action bar (Annotate / Reload / Compile / Open Blueprint)
  • Prompt preview modal
Shouldn't contain
  • Operational queue / readiness / coverage
  • Build state machine
  • System topology
  • Direct execution buttons (always hand off to Blueprint)

Blueprint · /portal/project/blueprint

Should contain
  • The 6 steps. Only the 6 steps.
  • Step 1: Context (from Cory's next_action OR Critique handoff)
  • Step 2: Task (files + acceptance criteria)
  • Step 3: Prompt (copy-ready dark code block)
  • Step 4: Execute (validation report textarea + Submit)
  • Step 5: Verify (pass/fail + project health)
  • Step 6: Iterate (Complete & continue / Back to Critique / Open Home)
Shouldn't contain
  • Capability grid / build buttons per card
  • Mode toggles (Manual/Autonomous)
  • Project setup wizard
  • System architecture viewer
  • Demo overlay
  • Local prioritization or recommendation logic
  • Beta banners / promotional content

Verify · (today: embedded in Blueprint Step 5)

Should contain (when extracted)
  • Pass/fail confirmation
  • Screenshot diff (before / after)
  • Acceptance criteria checklist (carried over from prompt)
  • "Mark verified" + "Reject" actions
  • Recent verifications history
Shouldn't contain
  • Audit / governance language
  • Compliance workflows
  • Test runner UI
  • Code diffs (those live in Critique re-render)

System · /portal/project/system-v2

Should contain
  • Tabs: Components · Architecture · Database · BPs · Operations (advanced) · Cognition (advanced)
  • Topology graphs (read-only)
  • Capability grid (no build buttons — link to Blueprint instead)
  • Dependency relationships
  • Advanced operator dashboards (collapsed by default)
Shouldn't contain
  • "What should I do next?" recommendations
  • Build buttons
  • Operational queue
  • Authority calls-to-action
  • Phase numbers in default-tab content
