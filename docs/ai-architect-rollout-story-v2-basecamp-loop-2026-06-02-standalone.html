Visual story #2 · The Basecamp loop · For Ali · 2026-06-02

Basecamp is the work queue.
Claude Code is the agent.
The loop closes every time.

You asked for a richer picture: how Akiwam (or anyone) actually operates with Claude Code while Basecamp tracks the work. This document shows two scenarios in detail, then explains the deeper architecture you described — every person has a personal Basecamp, every task becomes a record, every record becomes the raw material for the next skill. Once the org runs this way, it generates its own training set.

Chapter 1 · The three roles in every interaction

Every workflow has these three actors. Always the same three.

The picture is simpler than it sounds. There is a HUMAN (Akiwam). There is an AI (her Claude Code, scoped to her role). There is a TICKET (the Basecamp record that holds the work). The loop runs through all three, every time.

👤
Akiwam
The human. Owns the goal, the judgment calls, the final approvals. Tells Claude Code what to do in plain English. Verifies output before it ships.
🤖
Akiwam's Claude Code
The agent. Scoped to her role's tools + skills. Reads Basecamp tickets, gathers context, drafts output, posts back. Asks for confirmation on anything write-heavy or external.
📋
Akiwam's Basecamp project
The record. Holds every task she owns or AI-completes. Becomes the durable trace of her work + the mining ground for new skills.
The three roles never collapse. The human never directly touches the tool (Bonfire, Calendar, Gmail) — she talks to her Claude Code. Claude Code never executes without confirmation on irreversible actions. The Basecamp ticket holds the truth so future-Akiwam (or another person) can replay it later.
Chapter 2 · The loop, drawn once

Six steps. Top-left to bottom-right. Read it as a clock.

ALERT FIRES

Akiwam's system pings her

3
SHE INVOKES

"Claude, this needs XYZ"

Akiwam opens her Claude Code, references the ticket, tells it what to do in her own words. Plain English. Her judgment, the agent's hands.
↓ ↓ ↓
6
SKILL CANDIDATE

Becomes future skill

The completed loop is auto-tagged as a skill candidate. Ali sees it in his weekly queue and approves (or not) for the company library.
5
POSTED BACK

Ticket updated + closed

Claude Code posts the deliverable as a comment on the BC ticket. Akiwam reviews + checks it off. The ticket is now the record of the work.
4
CLAUDE EXECUTES

Reads BC + acts

Claude Code reads the ticket + related context, calls the tools (Calendar, Gmail, Bonfire), drafts output, asks Akiwam to confirm anything external.
Read the loop in one line: a task in Basecamp creates an alert, Akiwam tells Claude what to do, Claude does it, the ticket gets updated, the workflow becomes a candidate skill.
Chapter 3 · Scenario 1

The AI built half the work. Akiwam finishes it.

Ali approved an AI to draft a Colaberry landing page for a new utility-sector pilot. The AI produced a working page — but only the raw text. No layout, no UX, no PDF. Akiwam picks up the verification + the polish.

1
Landing page from text to client-ready PDF
Akiwam · her Claude Code · Basecamp
10:14 AM
Basecamp
A new ticket lands on Akiwam's Personal BC: "[REVIEW NEEDED] AI drafted utility-sector landing copy. Polish + format. Ship as PDF." Auto-tagged. AI has already posted its raw text in the description.
10:16 AM
Akiwam
Sees the alert in her Claude Code chat ("@you have 1 new task requiring verification"). Opens it.
10:17 AM
Akiwam
Reads the AI's raw text. It is good content but no design. She types: "@Claude — read ticket 9XXX, take this text, format it like our standard utility pre-read (navy + cream + red accent, exec-style typography), pull in the Colaberry logo + the 5-platform tile from the co-op ad, finalize as a 2-page PDF. Use the About Colaberry v6 layout as the template."
10:17 AM
Claude (logs)
Records the instruction. Begins fetching: reads BC ticket 9XXX, finds the About Colaberry v6 template in repo, locates the Colaberry logo + co-op ad tile, opens the standard utility pre-read CSS.
10:18 AM
Claude
Generates a 2-page HTML using the standard navy/cream/red theme. Renders to PDF via Playwright. Shows Akiwam a preview in the terminal.
10:20 AM
Akiwam
Reviews the PDF. Asks for one change: "Add a footer with David Lahme's contact info — pull from his vCard."
10:21 AM
Claude
Re-renders with the footer. New PDF ready. Akiwam approves.
10:22 AM
Hook
"About to post the final PDF as a comment on BC ticket 9XXX + check off the ticket. Confirm 'yes' + one-line reason."
10:22 AM
Akiwam
"Polished + reviewed, ready to ship. yes."
10:23 AM

Look at the rhythm. AI did the first half (text generation). Akiwam did the second half (verification + design judgment + polish). Claude Code did the boring middle (template lookup, rendering, posting). The ticket holds the record. The workflow is now a candidate skill — next time, "polish AI landing text to PDF" becomes a one-line slash command anyone on the team can use.

Chapter 5 · Personal Basecamp projects

Every person has a Basecamp project that is their work history.

The deeper architectural move: each employee has their own Basecamp project. Every task they own, every AI-completed task they verify, every conversation they have with their Claude Code — lands as a ticket or comment in their personal project. The project becomes the durable record + the mining ground for skills.

Why this matters: when Akiwam onboards a new intern next month, that intern reads Akiwam's BC project to learn the rhythm. When Ali wants to know "what skills has Kes been building?" he scans Kes's project. When someone leaves, their project + their skills stay. Yohan's "skills are the organization" answer from the Alden talk - this is how we operationalize it.

Six months from now you can replay any decision. Twelve months from now you have a 30-person org's worth of training data, organized by domain.
Chapter 6 · The skill extraction pipeline

From a single completed ticket to a company-wide capability.

Here is the path a one-time workflow takes to become a "Colaberry approved" skill in the library — the thing every employee can then slash-invoke.

Stage 1

Completed ticket

"Schedule deep-dive with Jen" — done by Akiwam, logged in her BC.
→
Stage 2

Auto-tag as candidate

Claude detects a repeatable pattern. Tags + saves the instruction template.
→
Stage 3

Ali weekly review

Friday 30-min: Ali sees the queue, approves or rejects each candidate.
→
Stage 4

Colaberry approved → library

Skill is committed to the AI Project Architect repo. Auto-syncs to all 4 employees' Claude Code overnight.
The math: if each pilot person produces 2 skill candidates per week, and Ali approves half, that is 4 skills/week joining the company library. After 90 days, the library has ~50 reusable skills the whole team can invoke. That is the compounding curve. Year-one ends with the org having codified its own playbook.
