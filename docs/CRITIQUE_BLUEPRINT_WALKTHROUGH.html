The mental model — which tab does what

The two surfaces are halves of one loop. Critique is where you see what's wrong. Blueprint is where you fix it with Claude Code. They hand off to each other through a single compiled prompt.

TabWhat it's forWhat you do hereWhat you walk away with
CRITIQUE Visually reviewing pages and pinning issues Open a page in an iframe, click Annotate, drop pins on problem areas, describe each issue, then click Compile prompt. A Claude Code prompt that bundles every pin into one structured ask, with verification steps.
BLUEPRINT Executing one focused build with Claude Code Walk through 6 stages: Context → Task → Prompt → Execute → Verify → Iterate. Copy the prompt, run it in Claude Code, paste the validation report back. A shipped change, verified against telemetry.

The handoff: when you click Send to Blueprint in Critique, the compiled prompt lands in Blueprint's Stage 3, the context bar says "Critique pending: /your-route", and you copy + run in Claude Code.

Fix found along the way

FIXCritique landing page defaulted to http://localhost:8888

Walking through Use Case 1 surfaced an immediate breakage. The "Start a new review" form's Preview origin defaulted to http://localhost:8888 on production — a stale dev URL that any operator typing a route into would have been blocked by mixed-content rules (HTTPS portal trying to load HTTP iframe). The hardcoded default in SessionPickerEmpty.tsx was replaced with window.location.origin, so the form now pre-fills https://enterprise.colaberry.ai in production, and your local dev origin in dev. Shipped as commit 6298c76 before the rest of the walkthrough was captured — so every screenshot below shows the corrected behavior.

CRITIQUEHow to use it — step by step

Critique is your visual review desk. Open any page in your portal (public site, admin surface, anything that loads in an iframe of the same origin), pin issues you can see, and let the system compile your pins into a Claude Code prompt. Two real use cases below.

BLUEPRINTHow to use it — step by step

Blueprint is the execution surface. Cory decides what's next; this page walks you through running it. Six stages — Context, Task, Prompt, Execute, Verify, Iterate — render in sequence. Two ways to land here: directly (Cory's next action takes priority), or via the Send to Blueprint button from Critique (a handoff prompt takes priority over Cory's next action).

1Use Case — Execute Cory's next action (no Critique handoff)

Scenario: You open Blueprint with nothing in flight from Critique. Cory has decided the next operational priority is to create an artifact for GET /api/courses (requirement REQ-027 has no artifact yet). You walk through the 6 stages, copy the prompt, run it in Claude Code, paste the validation report back, and verify.
1Open Blueprint
Click Blueprint in the top nav. The page reads "Execute the next step." The whisper line in the context bar tells you what to do next: "Click Mark accepted to track this as your active build."

2Read Stage 1 (Context) + Stage 2 (Task)
Context tells you what we're executing and why. Task gives the specifics — action type, requirement key, suggested files. These two stages set the stage; nothing for you to do here except read.
Stages 1 + 2 — Context (why) and Task (specifics)
Notice: Cory tags this with confidence 90%, blast radius low, est. 25m. That's not a guess — it's pulled from the same unified state every other surface reads.
3Stage 3 — the Claude Code prompt
A fully-formed Claude Code prompt with sections for Why, Requirement, Implementation expectations, and Verification. Three actions: Copy prompt, Open in new tab, Mark accepted (tracks this as your active build on Home).

After you run the prompt in Claude Code, paste whatever Claude reports back (the validation summary, tsc results, what shipped) into the textarea. Click Submit & verify. The verification reads telemetry — not just the report's claims — so the system knows whether the change actually landed.

A small project-health pill (here: 60%) and a recent-verifications counter. This is the "did it actually move the needle" sanity check — not an audit. If it didn't, you iterate.

Three exits: Complete & continue (mark this build done and refresh the next), Back to Critique (return to pin more issues), Open Home (back to the L1 authority view). The "Need the legacy build surface?" link is preserved for rollback.

Common gotchas + tips

Critique

  • The iframe is sandboxed. Some interactions inside the iframe (logins on third-party pages, popups) won't work. Stick to same-origin portal routes for full fidelity.
  • One session per page route. Each "Open visual workspace" creates a new session in the backend. Use the "Recent reviews" list to resume an existing one rather than spawning new ones.
  • Description quality drives prompt quality. "This looks bad" → bad prompt. "Hero headline X reads feature-y; rewrite to name the buyer's situation, then position the tiers as outcomes" → useful prompt. Be the engineer's brief.
  • Compile when you have 1-5 pins. The compiled prompt scales but tight bundles ship faster. If you have 12 pins, split into 2-3 compiles by theme.

Blueprint

  • Cory decides what's next, Blueprint executes. If you disagree with the priority, change it on Home (Cory authority surface) — not here. This page never argues with Cory.
  • "Mark accepted" tracks this as your active build on Home. Useful when you'll be away from your machine and want the next visit to resume right where you left off.
  • Validation reads telemetry. Stage 4 doesn't just trust the report you paste — it cross-references the project's actual state. Don't try to gaslight it.
  • Critique handoffs override Cory's next action in the prompt slot only. Cory's queue is unchanged; only this Blueprint session pulls from sessionStorage.

The loop

  • Critique → Blueprint is the visual-improvement loop. See it → pin it → compile → run in Claude Code → verify. Stay in the loop until the change is real, not just intended.
  • Critique without a Blueprint handoff is a half-loop. You can copy a compiled prompt and paste it manually into Claude Code, but you lose the verification trail. Prefer "Send to Blueprint" for anything you actually intend to ship.
  • Don't pin what isn't visible. Critique is for things you can see on the page. For "the API is wrong" type issues, work from Cory's queue on Home instead.
