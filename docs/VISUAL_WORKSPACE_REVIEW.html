03 UX walkthrough

What the user actually does, end to end.
  1. Navigate to Critique. Click Critique in the navbar.
  2. Pick or create. Calm landing screen with two options: pick from recent reviews, or open a new one with a target preview origin + page route.
  3. Three-pane shell loads. Sidebar (left) · iframe stage (center) · details (right) · sticky action bar (bottom).
  4. Click "Annotate". Annotate-mode badge shows in the stage; cursor becomes a crosshair; iframe pointer events are suppressed so clicks register on the overlay.
  5. Click anywhere on the page. A modal opens prefilled with the pin's normalized x,y coordinates.
  6. Fill in the issue. Title, kind, severity, what's wrong, expected outcome. Optionally click "AI suggest" to pre-fill from the curated pattern bank.
  7. Save. Backend persists the critique with the region. A numbered pin renders on the stage; the issue appears in the sidebar.
  8. Pick a pin. Click any pin to select it; the right panel shows the full critique + AI suggestions + decision controls.
  9. Accept suggestions. Each suggestion gets Accept / Reject / Defer. Accepted suggestions surface in the sidebar's "Ready for prompt" section.
  10. Compile. Click "Compile prompt" in the action bar. The backend's generate-prompt runs; if it returns empty, the local compiler builds the Markdown locally.
  11. Preview, copy, open Build Center. Compiled prompt opens in a modal with copy button, "open in new tab", and "Send to Build Center" action.

05 Annotation flow

Lightweight V1: click → modal → save. No advanced collaborative editing, no DOM mutation.
  1. User clicks Annotate on the action bar; annotateMode = true.
  2. Iframe pointer-events suppressed; transparent overlay captures all clicks.
  3. Crosshair guide lines follow the cursor for precise placement.
  4. Click drops a normalized (x,y) pin coordinate (0..1 of stage width/height).
  5. AnnotationModal opens with the pin coordinates pre-filled in a read-only hint footer.
  6. User fills title (auto-derived from description if blank), kind, severity, description, expected outcome, optional CSS selector.
  7. Optional: click AI suggest to pre-fill from critiquePatterns.ts by selected kind.
  8. Save calls session.addCritique({ kind, severity, description, region: {x,y,0,0}, target_selector, expected_outcome }).
  9. Pin renders on the stage with severity colour; sidebar list updates; annotate mode auto-exits.

Pin colour key

  • high — red pin
  • medium — amber pin
  • low — blue pin
  • resolved — green pin with check
  • Active pin gets a primary-colour ring

06 AI critique system

V1 ships a curated suggestion bank, not autonomous AI. Backend's existing generate-suggestions flow remains the canonical source for accepted suggestions.

Two layers

  1. Modal pre-fill (frontend-only). The "AI suggest" button in the annotation modal reads from critiquePatterns.ts — a curated bank of 1–2 hints per critique kind. Cycles through alternates on repeat clicks. Zero cost, zero network call. Used to give the user a strong starting point instead of a blank textarea.
  2. Backend suggestions (canonical). When a critique is saved, the backend's existing flow attaches AI-generated suggestions (these were already wired pre-V1; the workspace just renders them). Each suggestion is Accept / Reject / Defer; accepted suggestions feed the prompt compiler.

Suggestion bank coverage

spacing alignment color typography interaction accessibility hierarchy responsiveness workflow copy

All 10 critique kinds covered with at least one starter pattern. Easy to extend — add to the bank and the modal picks it up.

07 Prompt compiler

Backend-first with local fallback. The compiled output is production-grade Markdown ready for Claude Code.

Order of operations

  1. User clicks Compile prompt (action bar) or Compile prompt for this issue (right panel).
  2. Frontend calls session.generatePrompt() — the backend's existing endpoint that pulls accepted suggestions, file context, etc.
  3. If backend returns a non-empty prompt, use it (source: backend compiled).
  4. Otherwise, the local compilePromptLocally fallback builds a structured Markdown doc from in-memory critiques + suggestions + decisions (source: local fallback).
  5. Result opens in PromptPreviewModal with copy / open-in-new-tab / send-to-Build-Center actions.

Local prompt structure

  • Header: target page, preview URL, critique count.
  • Objective: address the pinned critiques; coordinates explained.
  • Critiques: numbered list with kind/severity badges, selector (if any), normalized region, issue text, expected outcome, accepted suggestions.
  • Implementation expectations: minimum-viable change, preserve contracts, use tokens, viewport check at 1280/1024/768.
  • Verification: tsc, visual check, BuildManifest emission, PROGRESS.md update.
  • Acceptance criteria: per-critique checkbox list of expected outcomes.

08 Build Center routing

Visual Workspace never builds directly. It hands off to Build Center via sessionStorage + navigation.

V1 mechanism

  • Storage key: visualWorkspace:pendingBuildPrompt — the compiled prompt text.
  • Storage key: visualWorkspace:pendingBuildSourceRoute — the page route the prompt came from.
  • Navigation target (V1): /portal/project/blueprint?build=visual-workspace — the System Blueprint page; Build Center as its own route is roadmap P3.
  • Future (P3): swap target to /portal/build; Build Center reads from the same storage keys.

Why sessionStorage and not URL params? Prompts can be long (thousands of chars), URLs choke at ~2k. sessionStorage is per-tab and cleared on close — no leakage to other tabs, no persistence past the session.

13 Out of scope (V1)

Explicit deferrals so V1 stays calm and shippable.

Not in V1

  • Pin drag / resize. V1 pins are point-only (width=height=0). Region rectangles are stored but not drawn.
  • DOM introspection inside the iframe. Cross-origin would block; V1 stays at the overlay level.
  • Real-time collaboration. Single-user only; no presence, no live cursors.
  • Toast notifications. One alert() for verification handoff; no toast library added.
  • Build Center as its own route. Roadmap P3. V1 routes to Blueprint with a flag + sessionStorage payload.
  • Verification Workspace. Roadmap P4. V1 stub writes to sessionStorage.
  • Component / route registry lookup at compile time. Roadmap P5 (UnifiedProjectState). V1 prompt names the page route only.
  • Pin auto-resolution. No detection of "this critique is fixed" after rebuild — user clicks Mark resolved manually.
  • Mobile layout. Sidebar + details collapse on < 768px; full functionality requires desktop.
  • Multi-page sessions. One session = one page. Switching route closes + reopens.

14 Critique categories & what to look for

When you open the V1, here's what to evaluate. Score yourself: would the platform speed your next QA loop?

Visual trust

  • Does the empty state feel calm, or empty?
  • Does the active session feel premium, or busy?
  • Is the iframe size appropriate, or do you wish it were bigger?
  • Are the pins distinguishable enough at a glance?

Annotate ergonomics

  • Does the cursor feel right in annotate mode?
  • Is the modal too long? Too short? Right fields?
  • Does "AI suggest" feel useful, or gimmicky?

Compile + handoff

  • Is the compiled prompt actually paste-ready, or does it need editing first?
  • Does the Build Center handoff feel direct enough?
  • Should the prompt preview show a diff or comparison view?

Sidebar utility

  • Are the 5 sections the right cuts?
  • Does the count badge add value?
  • Should "Resolved" be hidden by default?

15 Open questions for the user

Decisions that need your call before V2.
  1. Same-origin embedding. Is it OK that V1 only works for pages on the portal origin (or whichever origin is set as "Preview origin")? External pages will not embed due to X-Frame-Options. Worth adding a server-side puppeteer screenshot fallback in V2?
  2. AI critique source. V1 uses a curated bank. Is this enough, or should we route through OpenAI for the modal pre-fill too?
  3. Build Center landing. V1 lands on Blueprint with a flag. Should we ship a tiny dedicated "Pending build" page now (a few hours) or wait for the full Build Center in P3?
  4. Verification stub. V1 alerts and stores. Should we surface a "queued" pill in the navbar with a count, or wait for P4?
  5. Critique navigation. The new "Critique" navItem sits between V2 and Sessions. Is that the right place, or should it be the first item?
  6. Existing VisualReviewWorkspace. The legacy frontend/src/pages/project/VisualReviewWorkspace.tsx is still routed (none of its routes were touched). Deprecate it now, or wait until V1 is validated in production?
