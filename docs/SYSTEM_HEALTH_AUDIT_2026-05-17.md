## Follow-up dives

### Dive 1 — What `verified_partial` actually means

Sampled 3 of 120 `verified_partial` requirements:

| requirement_key | semantic_status | semantic_confidence | interpretation |
| --- | --- | --- | --- |
| `AUTO-1473E4DD-ANOMALY-ALERTS` | `semantic_not_aligned` | 0.90 | high confidence the code does NOT implement |
| `AUTO-1473E4DD-USER-EVENT-TRACKING` | `semantic_not_aligned` | 0.20 | low confidence, but still negative |
| `AUTO-66C7998C-ANOMALY-ALERTS` | `semantic_not_aligned` | 0.90 | high confidence not implemented |

**Implication:** the `verified_partial` label has nothing to do with "this is partially implemented." It's the merge case when the rule-based and semantic verifiers disagree. The semantic verifier is doing real work and producing real verdicts (most negative) — finding #1 from this morning (verifier blind to inline route handlers) is more nuanced: the verifier DOES analyze the code, it just can't recognize this particular pattern.

### Dive 2 — Warroom missing sections (false alarm)

The check 16 warning ("2/6 sections present") was caused by my naive expectation of section names. The actual top-level keys returned by `GET /api/portal/project/warroom`:

```
progress, current_action, requirements, coverage_summary,
recent_activity, artifact_graph, risk_summary
```

I had asserted on `next_action / verification / graph / risk` — those don't exist. The endpoint is healthy; my test was wrong. Harness patched ([scripts/systemHealthCheck.js](scripts/systemHealthCheck.js) check 16).

### Dive 3 — Secret-scan test inconclusive

Three attempts to trigger the validator's secret-pattern rejection were all blocked by the upstream Zod UUID format check (the validator wants RFC v4 format with proper version bits in the 13th hex). The secret scanner is documented to exist (CLAUDE.md §"No secrets in manifests") but I couldn't isolate it via this probe. A real test needs either (a) a valid v4 UUID + the secret pattern in a payload field that survives schema validation, or (b) a direct unit test against the scanner function. Logging for follow-up; not a current bug indicator.
