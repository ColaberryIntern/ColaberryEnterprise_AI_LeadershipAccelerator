The AI Operations
Command Center.

The fastest path to "what should happen next" for every ticket in the company. This document presents the full system architecture before any code is written. 15 deliverables. One mental model. One lean MVP recommendation at the end that delivers 80% of the value in two weeks.

Principal Product Architect · UX Lead · Staff Engineer · Workflow Automation Expert · AI Operations Designer
Prelude · The reframe

This is not a tool. This is the operating system.

Basecamp tracks tasks. Cory writes briefings. Inbox COS sorts mail. Each is useful in isolation. None of them answer the question that actually matters every morning - "what is the single fastest path through everything that has to happen today?" That question lives between systems, in the white space, and is currently answered by you holding it in your head.

The Command Center is the layer that owns that question. It sits on top of Basecamp (which stays the system of record), reads everything that moves, scores everything by urgency, identifies what AI can finish, what only you can decide, what is blocked and why, and what should be turned into a permanent skill. It is the cockpit. Basecamp is the engine.

Optimize for one number: average "waiting on human" hours per ticket. Everything else - throughput, automation count, skills compounded, hours saved - falls out of driving that single metric to zero.
Part I · Deliverable 1 · System Architecture

The six-layer stack, top to bottom.

Six layers, each isolated, each replaceable. The dependency graph runs top-to-bottom and never sideways - the Presentation layer never talks to BC directly, the Workflow layer never invokes the LLM directly. Strict layering keeps the system testable.

Layer 6 · Presentation

Command Center UI + mobile approval queue + Run-My-Day mode

React + Tailwind. Three-column desktop view + single-column mobile approval queue. Server-sent events for real-time updates. No business logic here - the UI only reads + emits intent.
React · Tailwind · SSE · React Query · Recharts
Layer 5 · Workflow

Approval router + automation engine + revision workspace

Stateful. Holds the approval queue, runs the automation triggers, manages multi-step continuations. Every "approve / revise / reject" comes through here. Every "on-approve, fire next step" comes from here.
Node.js · PostgreSQL · node-cron · BullMQ-lite
Layer 4 · Intelligence

Priority engine + AI Opportunity engine + Brand engine + Skill extractor + Meeting eliminator

Five scoring engines that run on every new event. Each produces a numerical score (0-100) + a reasoning JSON. Engines are stateless functions of (ticket context + history). Cheap models for triage, expensive only when reasoning matters.
OpenAI · Anthropic · prompt cache · rubric library
Layer 3 · Agents

Eight named agents - one per job

Triage / Priority / AI-Opportunity / Brand / Skill Extraction / Communication / Meeting Eliminator / Chief of Staff. Each agent is a single Node module with a typed input + typed output + cost ceiling. They never call each other - the Workflow layer orchestrates.
LangChain-free · MCP-shaped · cost-aware
Layer 2 · Sync

Continuous mirror of every source of truth into a local read-model

BC API every 2 min (with webhook backup). Gmail OAuth every 5 min. GitHub webhooks for skill commits. Calendar OAuth every 10 min. Sync workers never mutate the upstream - they only read + index. Push-back is the Workflow layer's job.
node-cron · PostgreSQL · Redis pub/sub
Layer 1 · Data sources

Basecamp · Gmail · GitHub · Calendar · CCPP · existing prod backend

The external sources of truth. Basecamp remains canonical for project state. Gmail remains canonical for communication. GitHub remains canonical for code + approved skills. We never duplicate state - we mirror it for read speed + score it for intelligence.
Basecamp 3 API · Gmail API · GitHub API · Google Calendar API
The architecture's load-bearing decision is that nothing in the system mutates an upstream source of truth except the Workflow layer. That single constraint is what lets us add capabilities without ever wrecking the underlying records.

Part III · Deliverable 3 · Agent architecture

Eight agents. Three model tiers. Strict cost accounting.

Each agent is a Node module exporting a single function with typed input + typed output. Agents never call each other. They are pure functions of (context, prompt, model) → (result, confidence, cost). The Workflow layer orchestrates.

Triage Agent
CLAUDE HAIKU · $
Classify every new BC todo into one of 5 buckets: human-required / AI-can-finish / AI-can-prepare / can-be-eliminated / wait-for-dependency. Sub-second response time. Runs on every new event.
in: {todo, recent_comments}
out: {category, confidence, reasoning}
Priority Agent
CLAUDE HAIKU · $
Score urgency 0-100 from rubric: due date, revenue impact, customer impact, dependencies, blocked downstream, waiting time, contract value, executive visibility, legal/compliance. Runs daily on every open todo + on any state change.
in: {todo, project_context, downstream_blocked_count}
out: {urgency_score, factor_breakdown}
AI Opportunity Agent
GPT-4O-MINI · $$
For each todo: can AI finish / partially finish / prepare / eliminate? Produces concrete recommended action + estimated time savings. Runs daily on every open todo.
in: {todo, available_skills, user_role}
out: {opportunity_score, recommended_action, estimated_savings_minutes}
Communication Agent
GPT-4O-MINI · $$
Draft emails, calendar invites, Basecamp comments, Slack messages, executive summaries, status updates, meeting agendas, decision memos. Triggered by user request or by automation engine.
in: {prompt, audience, tone, source_artifacts}
out: {draft_html, draft_text, suggested_subject, recipients}
Brand Compliance Agent
GPT-4O-MINI · $$
Review any visual or written artifact for brand alignment, visual quality, readability, executive quality, consistency. Returns Brand Score 0-100 + one-click fix list. Runs on every artifact before approval queue entry.
in: {artifact_html_or_image, brand_rubric_version}
out: {brand_score, factor_scores, one_click_fixes[]}
Meeting Elimination Agent
GPT-4O-MINI · $$
When a todo contains "meeting / call / discussion / review session," evaluate: can this be replaced by async work? Proposes specific alternative (email, decision doc, approval form, async collab) + estimated savings.
in: {meeting_todo, participants, prior_async_attempts}
out: {can_replace, alternative, estimated_savings_minutes}
Skill Extraction Agent
CLAUDE OPUS · $$$
After a workflow completes, examine: should this become a reusable skill? Pattern-match against existing skills + propose a skill definition with name, trigger, inputs, outputs, required approvals. Outputs candidate for Ali's weekly review queue.
in: {workflow_run, related_completed_todos}
out: {should_extract, skill_definition (YAML), confidence}
Chief of Staff Agent
CLAUDE OPUS · $$$
Synthesizes org-wide state into "Run My Day" view. Top 5 decisions / 5 approvals / 5 bottlenecks / 5 opportunities / 5 automations. Runs at 7am daily + on-demand. The only agent that sees the whole org.
in: {full_org_state_snapshot}
out: {five_lists, expected_impact, total_review_minutes}
Budget envelope: at scale, we expect ~50,000 agent calls/day across all 8 agents. With aggressive tiering + caching, target spend: $120-180/day on LLM costs ($3.6-5.4K/month). The $90/$10 ratio from Alden's playbook puts our headcount AI budget at ~$3K/mo - we land inside.

Part V · Deliverable 5 · Basecamp integration

BC stays the source of truth. We mirror, score, push back.

The integration is bidirectional but asymmetric. We pull aggressively (every 2 min cron + webhooks where available) and mirror to our DB for fast querying. We push back narrowly - only through the Workflow layer, only as comments, todos, and uploads - never overwriting BC state directly.

DirectionWhatHowFrequency
PULLevents.json (all activity)BC API /events.json?since=TEvery 2 min cron
PULLProjects + todos + comments + uploadsBC API /projects/<id>.json etcOn event change
PULLWebhook for instant updatesBC webhook subscriptionReal-time
PUSHApproval results posted as commentsPOST /buckets/<b>/recordings/<r>/comments.jsonOn user approve
PUSHArtifacts uploaded to VaultPOST /attachments.json + Vault flowOn artifact creation
PUSHNew todos for follow-up workPOST /buckets/<b>/todolists/<l>/todos.jsonOn workflow continuation
PUSHMark todos completePUT /buckets/<b>/todos/<t>/completion.jsonOn final approval

Rate-limit strategy: BC's limit is 50 req/10s. The sync worker keeps a token bucket + queues + retries with backoff. The push side is gated through a single rate-limited service so multiple workflow continuations cannot trample BC.

↓

↓

Part VIII · Deliverable 8 · Automation workflow engine

Trigger → Condition → Actions. Pure event-driven.

Automations are typed rules that fire on events from the Sync layer. Each automation is a JSON document in the automations table. The engine subscribes to event streams, evaluates conditions, runs actions (often invoking agents). Designed to be the same data model Ali sees in the Command Center "Automations" sidebar.

Part XIII · Deliverable 13 · Risks identified

Six risks, each with a specific mitigation.

High · Operational

Over-automation eats decisions Ali should own

The Workflow layer takes actions without enough human visibility. A bad pattern compounds before anyone notices.
Mitigation: every "auto-run" tier has a confidence threshold (default 0.9) + a weekly retro that surfaces every auto-decision for Ali's review. A 5-minute audit on Fridays.
High · Cost

LLM bills spiral as agent calls grow

Phase 4 automation engine + 17 users + 50 daily skills could push to $500+/day in LLM if not tiered carefully.
Mitigation: per-day budget cap with auto-pause at 80%. Tiering: Claude Haiku for triage, GPT-4o-mini for routine drafts, Claude Opus reserved for the Chief of Staff + Skill Extraction agents.
Medium · Integration

BC API rate limits + state divergence

Our mirror drifts from BC truth. We push back stale data. Workflows fail mid-flight.
Mitigation: sync worker has reconciliation pass every hour (compare hash of local vs BC). Drift alerts page Ali. Push side uses a single rate-limit queue.
Medium · UX

Approval fatigue from too many queue items

If everything is "important," nothing is. Ali tunes out the panel and we are back to BC plus extra noise.
Mitigation: Run-My-Day caps daily at 5 + 5 + 5 + 5 + 5. Everything else queues. Auto-batch + Cory digest absorb the rest. Aggressive de-duplication.
Medium · Quality

Skill Extraction Agent hallucinates patterns

Marks one-off workflows as "reusable" and clutters the library with junk skills nobody uses.
Mitigation: usage tracking on every skill. Auto-deprecate skills not invoked in 90 days. Weekly review queue stays - Ali catches the bad ones.
Low · Trust

Brand Agent gives false-positive "non-compliant" flags

Routine artifacts get blocked at the Brand step, users disable the gate, brand drift returns.
Mitigation: Brand rubric is versioned; users can mark "false positive" inline. Brand Agent retrains on labeled feedback monthly. Confidence threshold stays high.
Part XIV · Deliverable 14 · Cost optimization

The biggest line item is LLM tokens. Here is how we keep it sane.

  1. Tiered model selection. Triage and Priority use Claude Haiku ($0.25/1M in, $1.25/1M out). Routine drafting (Communication, Brand, Meeting Eliminator) uses GPT-4o-mini ($0.15/1M in, $0.60/1M out). Chief of Staff + Skill Extraction use Claude Opus only when reasoning matters ($15/1M in, $75/1M out). Expected blended cost per call: $0.001-0.003 for tier 1, $0.005-0.015 for tier 2, $0.05-0.20 for tier 3.
  2. Aggressive caching. Daily Priority scores compute once + cache 24h. AI Opportunity scores re-compute only on state change. Brand Agent caches results by artifact hash. Anthropic prompt caching is enabled across all calls.
  3. Batch where possible. Score 20 todos in one Priority Agent call, not 20 separate calls. Reduce per-call overhead.
  4. No streaming for batch operations. Streaming costs slightly more in tokens and adds latency. Reserved for UI-facing interactions only.
  5. Per-employee daily cap. Hard limit per user (default $5/day). Soft alert at 80%. The cap is the safety net for runaway behavior.
  6. Skill cache. A skill's standard prompt + skeleton is cached. Only the user inputs vary, so per-invocation cost is mostly in caching.
  7. Drop redundant context. The BC walker truncates aggressively. We do not feed the LLM every comment ever - just the most relevant recent N.
  8. Self-host nothing yet. Even with the costs above, hosted Claude + OpenAI APIs come in well under $5K/month. Self-hosting a 70B model on the Hetzner VPS is a 2027 conversation, not now.
Expected monthly LLM spend at full Phase 5 scale (17 employees, 23+ active automations, ~50K agent calls/day): $3.6K - $5.4K/month. Inside Alden's $90/$10 ratio for a 30-person org budgeted at $30K/mo on payroll.
Part XV · Deliverable 15 · The leanest 80/20 architecture

One screen. One engine. Two weeks. 80% of the value.

If I could only build a single thing that delivers 80% of the promise — the thing that turns Basecamp from a tracking system into an AI-guided operations system — it is:

The 80/20 recommendation

The Waiting on Human panel + the rule-based Priority Engine.

That is it. No LLM-driven priority agent yet. No Brand Agent. No Skill Extraction yet. Just: mirror BC → rule-score each open todo for urgency → list them in a single sorted panel with one-click approve → on approve, post a comment + mark complete + advance to the next item.

Ali's day looks like this: open Command Center at 7am. See 8 items in the Waiting on Human panel, sorted by urgency. Each one shows: title, why it's urgent (downstream blocked count + due date), and an "Approve / Review" pair of buttons. Approve advances. Review opens the full BC ticket inline. 14 minutes later, the queue is clear and the org is unblocked.

Everything else - AI suggestions, Brand scoring, skill extraction, meeting elimination, Run My Day, automation engine, Chief of Staff synthesis - is additive and earns its place after the MVP proves the panel reduces "waiting on human" hours.

Why this is the cut

  • It does not need any AI to ship. Rule-based urgency is enough. Adding LLM scoring is a Phase 2 improvement, not a Phase 1 requirement.
  • It addresses the highest-leverage pain. "Waiting on human" is the bottleneck. The panel makes it visible + actionable in one place.
  • It is fully owned by us. No external integrations beyond BC + Postgres + a simple React frontend. Two weeks of work.
  • It produces immediate metrics. Day 1 ships with "approvals completed today" + "hours unblocked." Ali knows by day 7 whether this is working.
  • It does not require trust in agents yet. Every decision is still Ali's. The system just makes the queue legible. Trust in agents is earned over Phase 2.
The MVP test is binary: can Ali close his daily approval queue from one panel in under 15 minutes? If yes after two weeks, we earned the right to build Phase 2. If no, we diagnose first, build second.
