04 Real health score — wiring

The hardcoded 90 is gone. Health is now a real composite that moves with the project. 
// Was
function buildHealth(): HealthProfile {
  return { score: 90, regressions_24h: 0, verification_pass_rate: 1 };
}

// Is
function buildHealth(progress: any, recentActions: any[]): HealthProfile {
  let healthScore = 60;
  if (progress?.breakdown) {
    const githubPct   = pct(progress.breakdown.githubHealth?.score);
    const workflowPct = pct(progress.breakdown.workflowProgress?.score);
    healthScore = Math.round(githubPct * 0.5 + workflowPct * 0.5);
  }
  // Verification pass rate from recent NextAction lifecycle.
  let pass = 0, total = 0;
  for (const row of recentActions || []) {
    if (row.status === 'completed') { pass += 1; total += 1; }
    else if (row.status === 'dismissed' || row.status === 'expired') { total += 1; }
  }
  const verification_pass_rate = total > 0 ? pass / total : 1;
  return { score: clamp(healthScore), regressions_24h: 0, verification_pass_rate };
}
  • Two real signals. GitHub freshness + workflow progression both come from the existing progress engine — no new computation, no new dependency.
  • Real verification pass rate. Derived from NextAction lifecycle: completed counts as a pass, dismissed/expired count toward total. When there's no recent activity the rate defaults to 1 (no failures yet).
  • Future: regressions_24h still hardcoded 0; wire from the SystemStateEngine summary in the next sprint.

05 Governance recs in the unified queue

Pending governance recommendations now appear as ranked queue entries alongside Cory's actions.

How they enter the queue

  • Synthesizer adds loadPendingGovernanceRecs(enrollment_id) as a parallel source (top 5 by priority).
  • Each rec becomes a PriorityCandidate with source: 'governance_recommendation'.
  • The model uses priority: 1..99 where 1 = top; the engine inverts to raw_priority = 100 − priority so its higher-is-more-important convention holds.
  • blast_radius is derived from risk_level: high → high blast, elevated/moderate → medium, low → low.
  • time_est_minutes: 15 default (operator review SLA placeholder).

What the user sees

Cory Home's queue now mixes next_action rows with governance_recommendation rows, sorted purely by priority. The source label below each title says "Governance" for these rows so the operator knows the origin without leaving Home. useGovernanceRecommendations still renders independently anywhere it was already mounted; both surfaces will agree because both ultimately read the same model.

07 ProjectNextActionPanel migration diff

Same panel UI. Same accept/complete buttons. New data source. 
// Was
const [action, setAction] = useState<NextActionData | null>(null);
const [loading, setLoading] = useState(true);
const fetchAction = useCallback(() => {
  portalApi.get('/api/portal/project/next-action').then(...);
}, []);
useEffect(() => { fetchAction(); }, [fetchAction]);

// Is
const { state, loading, refresh } = useUnifiedProjectState();
const action: NextActionData | null = useMemo(() => {
  const next = state?.next_action;
  if (!next || next.source !== 'next_action') return null;
  return {
    id: next.source_id || '',
    title: next.title,
    action_type: (next.metadata?.action_type as string) || 'build_feature',
    reason: next.reason,
    priority_score: next.priority_score,
    confidence_score: next.confidence_score,
    status: 'pending',
    metadata: next.metadata as any,
  };
}, [state?.next_action]);

// Mutations: still POST to /accept and /complete; then refresh()
const handleAccept = async () => {
  await portalApi.post('/api/portal/project/next-action/accept', { action_id: action.id });
  await refresh();   // every other surface sees the new state immediately
};

The panel filters by source === 'next_action' so it only renders Cory-engine items, not governance recs (those are surfaced in the queue elsewhere, not in this single-action panel).

GuidedExecutionPanel

/**
 * GuidedExecutionPanel — step-by-step guidance renderer for an accepted Cory action.
 *
