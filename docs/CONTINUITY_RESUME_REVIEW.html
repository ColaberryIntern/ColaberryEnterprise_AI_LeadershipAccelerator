4Active-path priority cascade (logic diagram)

The continuation surface picks ONE row. The decision flow is intentionally linear — falls through to the next branch only if the prior didn't match.

state arrives → useActivePath(state, frozenMemory, sessionStorage signals) │ ▼ ┌─── state.active_build present? ────┐ YES → "Continue active build" [FRESH] │ │ NO └─→ /portal/project/blueprint ▼ ┌─── sessionStorage pendingBuildPrompt? ┐ YES → "Run compiled critique prompt" [FRESH] │ │ NO └─→ /portal/project/blueprint?build=visual-workspace ▼ ┌─── memory.lastDrawerOpen set? ───┐ YES → "Reopen {drawer} breakdown" [AMBIENT] │ │ NO └─→ /portal/home?drawer={drawer} ▼ ┌─── memory.lastSystemTab set? ────┐ YES → "Return to System · {tab} · BP {id}" [AMBIENT] │ │ NO └─→ /portal/project/system-v2?tab={tab} ▼ ┌─── lastCritiqueAt < 2h ago? ─────┐ YES → "Return to Critique · last session {N}m ago" [FRESH if <30m] │ │ NO └─→ /portal/visual-workspace ▼ ┌─── state.next_action exists AND memory.lastSnapshotAt set? ┐ YES → "Pick up next priority" │ │ NO └─→ action.target_route ▼ return null (card hides — no continuation to show)

Reading priority: top → bottom. First match wins. Each branch carries its own freshness tone — fresh branches get a green left-stripe + ws-delta-rise entrance; ambient branches stay primary-blue.
